# Fantasy Scene Writing — Architecture Decisions

---

## 决策 1：Scene = 最小写作单位

**层级结构**：Project → EP → Scene → Chapter（物理切分）

Scene 本身已足够小，引入子层级只会增加：衔接成本、多层 QC、流程复杂度。

**适用条件**：只有当满足以下全部时才考虑拆分：
1. 单个 Scene 超过 LLM 单次输出的 3 倍仍然写不完
2. Scene 之间有完全独立的不同 POV 需要强制分段
3. 确实存在「这个 Scene 的第 2 部分和第 3 部分之间需要插入另一个 Scene」的情况

---

## 决策 2：写作段一口写完，不拆分，不逐 Scene QC

**流程**：一口气写完所有 Scene → 合并输出 ep{N}.md → 全稿一次性 QC

| 问题 | 逐 Scene 写+QC | 一口气写完 |
|------|---------------|-----------|
| 衔接断层 | LLM 在 Scene 边界容易出现弧光回退或重复动作 | 同一上下文连续写完，弧光状态保持连贯 |
| QC 阻塞 | FAIL 导致流程中断，需要 retry，增加心理负担 | 所有问题 RECORD，不阻塞，流程顺畅 |
| 效率 | 每 Scene 都要创建 QC 任务 + 等 QC 完成 + 再创建下一写作任务 | 单次写作，单次 QC，节省 N 个任务创建开销 |

**适用条件**：
- Scene 总数 ≤ 10（超过 10 个建议拆 EP）
- 每个 Scene 设计文档已经完成且经过 QC

---

## 决策 3：QC 只 RECORD，不 FAIL

所有检查项 RECORD，不 FAIL，不阻塞。

写作质量判断本质上是主观的。让 QC 严格区分 PASS/FAIL 会造成：过度保守的阈值设定、阻塞真正有价值的创作、retry 消耗额外 API 调用。

**代价**：需要用户自己审阅 RECORD 报告判断是否需要重写。

---

## 决策 4：scene-anchor 不需要

一口写完，不需要跨 Scene 协调。scene-anchor 的作用是防止 LLM 在跨 Scene 写作时出现弧光回退。当写作段一口气完成时，这个问题消失。

---

## 设计原则总结

1. **最小粒度原则**：只在必要时才引入新层级。
2. **一口完成原则**：写作段越连续，弧光越连贯。拆分只在物理限制（LLM 输出上限）时才有意义。
3. **QC 协议化原则**：QC 是观察报告，不是守门员。只记录，不阻塞。
4. **无状态写作原则**：如果写作段能一口气完成，状态同步文件就不需要。

---

## 反模式警示

| 反模式 | 特征 | 代价 |
|--------|------|------|
| 过度拆分层级 | 「我觉得 Scene 太长了」→ 加子层 | 衔接成本、anchor 维护成本、多层 QC |
| 逐节点 QC | 每个设计/写作节点都有 FAIL 判断 | 流程中断、retry 消耗、心理负担 |
| 预状态同步 | 写之前先同步状态文件 | 对 LLM 跨节点记忆有信心，不需要外部状态文件 |
