# Fantasy Scene Writing — Architecture Decisions

---

## 决策 1：Scene = 最小写作单位

**层级结构**：Project → EP → Scene → Chapter（物理切分）

Scene 本身已足够小，引入子层级只会增加：衔接成本、多层 QC、流程复杂度。

**适用条件**：只有当满足以下全部时才考虑拆分：
1. 单个 Scene 超过 LLM 单次输出的 3 倍仍然写不完
2. Scene 之间有完全独立的不同 POV 需要强制分段
3. 确实存在「这个 Scene 的第 2 部分和第 3 部分之间需要插入另一个 Scene」的情况

---

## 决策 2：写作以 Scene 过程稿落盘，整章 QC 统一执行

**流程**：逐 Scene 写出 `ep{N}-scene{X}.md` → 全稿 QC 顺序读取全部 Scene 过程稿 → 用户确认收尾后生成 `ep{N}.md`

| 问题 | 直接维护可编辑合并稿 | Scene 过程稿 + 收尾合并 |
|------|--------------------|------------------------|
| 用户编辑一致性 | dashboard 上改的合并稿可能不是下游真实输入 | 用户改的就是 agent 真实消费文件 |
| 多 agent 分工 | 多个 worker 容易争抢同一合并文件 | 每个 worker 只写自己的 Scene 文件 |
| 收尾语义 | 容易把“AI 刚写完”误当成“已确认成稿” | 最终稿只在确认节点生成，语义清晰 |

**适用条件**：
- Scene 设计文档已经完成且经过 QC
- 需要让 dashboard、人工修改和多 agent 写集合保持一致

---

## 决策 3：QC 只 RECORD，不 FAIL

所有检查项 RECORD，不 FAIL，不阻塞。

写作质量判断本质上是主观的。让 QC 严格区分 PASS/FAIL 会造成：过度保守的阈值设定、阻塞真正有价值的创作、retry 消耗额外 API 调用。

**代价**：需要用户自己审阅 RECORD 报告判断是否需要重写。

---

## 决策 4：scene-anchor 仍然不是必需

虽然正文过程稿按 Scene 落盘，但真实状态同步仍然由：

- Spine
- Scene Design
- Write QC
- anchor-update-draft

共同承担。

不额外引入 scene-anchor 文件，避免把状态层再细分一层。

---

## 设计原则总结

1. **最小粒度原则**：只在必要时才引入新层级。
2. **真实文件原则**：dashboard 可编辑文件必须等于下游真实输入文件。
3. **QC 协议化原则**：QC 是观察报告，不是守门员。只记录，不阻塞。
4. **无状态写作原则**：如果写作段能一口气完成，状态同步文件就不需要。

---

## 反模式警示

| 反模式 | 特征 | 代价 |
|--------|------|------|
| 过度拆分层级 | 「我觉得 Scene 太长了」→ 加子层 | 衔接成本、anchor 维护成本、多层 QC |
| 逐节点 QC | 每个设计/写作节点都有 FAIL 判断 | 流程中断、retry 消耗、心理负担 |
| 假合并稿 | 过程阶段维护一个可编辑 merged draft | 用户误以为修改已生效，实际下游读取的是 Scene 文件 |
