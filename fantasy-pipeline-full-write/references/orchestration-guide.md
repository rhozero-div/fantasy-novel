# 调度实现注意事项

> 本文件只讨论如何把 fantasy pipeline 接到任意调度系统、任务系统或人工串行执行流程中。

---

## 定位

本文件不是协议正文。

真正的 canonical protocol 在 `../SKILL.md`，其中定义了：
- 阶段边界
- 必要输入与输出
- 哪些节点必须停下等用户确认
- 哪些衔接可以自动发生，哪些不可以

本文件只说明：如果你要做执行层，应该遵守哪些约束。

---

## 调度实现必须遵守的约束

任意执行系统都应保留以下保证：

1. **阶段内 QC 可以自动衔接**
   - Spine → spine-qc
   - Scene Design → design-qc
   - Write → write-qc

2. **QC 之后不得自动进入下一创作阶段**
   - spine-qc 之后：等用户确认再进入 Scene Design
   - design-qc 之后：等用户确认再进入 Write
   - write-qc 之后：等用户决定是否处理锚点更新草案

3. **write-qc 只生成 `anchor-update-draft.md`**
   不得直接修改全局锚点文件。

4. **进入 `ignite EP{N+1}` 前**
   如果上一 EP 仍有 pending `anchor-update-draft.md`，必须先结清。

---

## 调度层的最小职责

一个可用的执行层至少应能：

- 为每个阶段调用正确的专项 skill
- 通过文件路径把上游产物传给下游
- 检查阶段输出文件是否已生成
- 在创作产物存在后运行该阶段对应的 QC
- 在定义好的确认节点停下，等待用户决定
- 在锚点草案未结清时阻止进入下一 EP

---

## 可移植性建议

为了保持实现可移植：

- 不把某一种任务系统、队列系统或 worker 产品写进主协议
- 把文件产物视为阶段之间最稳定的接口
- 把重试、心跳、任务 ID、派发语义等实现细节留在执行层
- 如果要记录某个具体实现，明确标注它只是示例，不是协议本体
